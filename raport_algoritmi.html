§0Arhitectura generală — 8 etape

Pipeline-ul aplicației procesează o factură PDF în 8 etape, fiecare cu propriul scop. Etapele 5–6 rulează în paralel pentru ambele firme.

§1Decizia hibridă text-vs-OCR

Locație: src/pdf_processor.py :: is_text_sufficient()

După extragerea textului nativ cu PyMuPDF, aplicația decide dacă merită apelată OCR-ul Tesseract sau direct LLM-ul. Algoritmul cere simultan ambele condiții — un singur prag de lungime nu e suficient, pentru că o factură scanată poate produce 200 de caractere de „zgomot" fără sens semantic.

def is_text_sufficient(text: str) -> bool:
    if not text or len(text.strip()) < MIN_TEXT_LENGTH:   # 200 chars
        return False
    lowered = text.lower()
    return any(keyword in lowered for keyword in INVOICE_KEYWORDS)

INVOICE_KEYWORDS e o listă bilingvă RO+EN: factura, factură, invoice, total, tva, vat, cui, subtotal, iban, furnizor, supplier, client, customer. E suficient să apară unul.

  text ──> len(text) >= 200?  ──┐
                                 │
                                 ▼
                        contine keyword?  ──> true: trimite direct la LLM
                                 │
                                 ▼
                              false: rendereaza PNG → Tesseract → LLM

§2Prompting constrâns la JSON

Locație: src/local_extractor.py :: _run_ollama()

LLM-ul local primește 3 mecanisme suprapuse de constrângere care minimizează halucinația și forțează output JSON valid:

MecanismCum funcționează
System prompt cu interdicție explicită„extract only information ACTUALLY present, do NOT infer or invent, use null for missing fields"
Schemă JSON impusă în promptStructura completă (Supplier/Customer/items/totals) e inserată ca template în user message
format="json" la OllamaLa nivel de tokenizer, Ollama forțează output sintactic JSON valid
temperature=0Determinism: același input → același output
După parsing, dacă output-ul nu poate fi decodat ca JSON valid, funcția parse_json_safe() încearcă să recupereze conținutul dintre primul { și ultimul }, eliminând eventuale fence-uri Markdown.

§3Normalizare CUI

Locație: src/company_verifier.py :: normalize_tax_id()

CUI-ul de pe factură poate apărea în multiple formate: RO12345678, ro 12345678, RO 12 34 56 78 . ANAF acceptă doar partea numerică. Algoritm în 2 pași:

def normalize_tax_id(tax_id):
    cleaned = re.sub(r"\s+", "", tax_id.strip())              # 1. fara spatii
    cleaned = re.sub(r"^RO", "", cleaned, flags=re.IGNORECASE) # 2. fara prefix RO
    return cleaned
Detaliu tehnic: am folosit flags=re.IGNORECASE ca argument separat, NU inline (?i). Python 3.14 nu mai permite flag-uri inline în mijlocul pattern-ului, doar la start.
InputOutput
"RO12345678""12345678"
"ro 12345678""12345678"
" 12345678 ""12345678"

§4Name matching slug-based

Locație: src/company_verifier.py :: compare_invoice_supplier_with_verified_data()

Compararea numelor de firmă trebuie să tolereze variațiile de formă legală: "SRL" vs "S.R.L." vs "S R L", diacritice etc. Algoritmul reduce ambele șiruri la slug-uri alfanumerice și compară prin incluziune.

def _slug(text):
    return re.sub(r"[^a-z0-9]", "", text.lower())

a = _slug("Test S.R.L.")     # "testsrl"
b = _slug("TEST SRL")        # "testsrl"
match = a == b or a in b or b in a   # True

Decizia a in b or b in a tratează cazul în care un nume e prefix sau sufix al celuilalt — ex. „Test SRL Bucuresti" vs „Test SRL" se consideră match.

§5Address matching prin token-overlap

Locație: src/company_verifier.py

Adresele sunt mai dificile — abrevieri (Str./Strada), ordinea diferită, diacritice. Folosim un algoritm Jaccard-like: setuim cuvintele cu lungime ≥ 3 caractere și verificăm overlap-ul.

a_tokens = set(re.findall(r"\w{3,}", inv_addr.lower()))
b_tokens = set(re.findall(r"\w{3,}", verification.address.lower()))
overlap = len(a_tokens & b_tokens)
threshold = max(1, min(len(a_tokens), len(b_tokens)) // 3)
match = overlap >= threshold

Pragul de cel puțin 1/3 din mărimea celui mai mic set permite o toleranță rezonabilă fără a accepta adrese complet diferite.

Adresă facturăAdresă oficialăOverlapMatch?
Str. Mare 5Strada Mare nr. 5{mare} = 1✓ da
Bd. Magheru 12Calea Victoriei 25{} = 0✗ nu

§6Detecția prefixului VAT-UE și auto-fallback

Locație: src/company_verifier.py :: verify_company()

Aplicația suportă firme din orice țară UE prin combinația ANAF (specific România) + VIES (toate țările UE). Dacă primary provider eșuează, sistemul încearcă automat alternativul, în funcție de prefixul VAT detectat.

  verify_company(name, tax_id)
       │
       ▼
  primary_provider == "anaf"?
       │
       ├── DA ──> ANAF call ──> success? ─── DA ──> return
       │             │                          NU
       │             ▼
       │       has EU VAT prefix?
       │             │
       │             ├── DA ──> VIES fallback ──> return
       │             │
       │             └── NU ──> return primary error
       │
       └── primary_provider == "vies" ──> ...similar logic

Detecția prefixului UE folosește un set de 28 coduri ISO de stat:

EU_VAT_PREFIXES = ("AT","BE","BG","CY","CZ","DE","DK","EE","EL","ES",

                   "NL","PL","PT","RO","SE","SI","SK","XI")

§7Clasificarea domeniilor pentru mențiuni

Locație: src/company_news_search.py :: _classify()

Rezultatele DuckDuckGo sunt etichetate în 4 categorii pentru a fi sortate și afișate diferit în UI: website (site propriu al firmei), registry (anaf, listafirme etc.), social (LinkedIn, Facebook), news (presă).

  url = "https://www.fulloutmedia.ro/contact"
       │
       ▼
  domain = "fulloutmedia.ro"
       │
       ▼
  ┌─ in REGISTRY_DOMAINS? ──> "registry"
  │
  ├─ in SOCIAL_DOMAINS? ────> "social"
  │
  ├─ slug(name) in domain? ─> "website"   ← caz în care e site propriu
  │
  └─ default ───────────────> "news"

Heuristica "slug-ul firmei e substring în partea registrabilă a domeniului" e cea care identifică automat site-ul propriu al firmei, chiar și fără o listă predefinită.

§8Filtrare conținut adult (2 niveluri)

Locație: src/company_news_search.py :: _is_adult_result()

Filtrarea se face în două straturi suprapuse, ca să asigure că rezultatele inadecvate nu intră într-un raport de verificare a firmei.

§10Paralelizare cu ThreadPoolExecutor

Locație: app.py :: main()

Verificarea celor două firme (supplier + customer) e I/O-bound — fiecare așteaptă răspunsul ANAF/VIES/DuckDuckGo. Rulându-le în paralel cu 2 fire, timpul total devine max(Tsupplier, Tcustomer) în loc de Tsupplier + Tcustomer.

with ThreadPoolExecutor(max_workers=2) as pool:
    sup_future = pool.submit(_audit_party, invoice.supplier)
    cus_future = pool.submit(_audit_party, invoice.customer)
    sup_v, sup_c, sup_n, sup_r = sup_future.result()
    cus_v, cus_c, cus_n, cus_r = cus_future.result()

Scădere de timp măsurată în testare: ~25s → ~13s end-to-end pentru o factură reală.

§11Generare XML deterministă

Locație: src/xml_generator.py :: generate_invoice_xml()

Generarea XML nu trece prin LLM. Datele validate de Pydantic devin un arbore ElementTree, apoi sunt serializate și pretty-printate cu minidom. Asta garantează:

  • Reproductibilitate bit-cu-bit (același JSON → același XML)
  • Escape automat al caracterelor speciale (< > &) — preîntâmpină XML injection
  • Performanță deterministă (zero ms latență AI)
root = ET.Element("Invoice")
_add_child(root, "InvoiceNumber", invoice.invoice_number)
_build_supplier(root, invoice.supplier)
_build_customer(root, invoice.customer)
items_node = ET.SubElement(root, "Items")
for item in invoice.items:
    _build_item(items_node, item)
_build_totals(root, invoice.totals)
_build_company_verification(root, "SupplierVerification", ...)
_build_company_verification(root, "CustomerVerification", ...)
raw = ET.tostring(root, encoding="utf-8")
pretty = minidom.parseString(raw).toprettyxml(indent="  ")

safe_text(value) tratează uniform None → "", float → format optim cu trailing zeros stripped, etc.

§12Coercer Pydantic — OptStr / OptFloat

Locație: src/schema.py

LLM-urile mici (Llama 3.2 3B) câteodată returnează [] sau {} pentru câmpuri care ar fi trebuit null, sau string-uri precum "1.234,56 RON" pentru numere. Tradițional, Pydantic ar fi căzut pe ValidationError. Soluția: două tipuri custom OptStr și OptFloat cu validator BeforeValidator care normalizează input-ul.

OptStr   = Annotated[Optional[str],   BeforeValidator(_to_opt_str)]
OptFloat = Annotated[Optional[float], BeforeValidator(_to_opt_float)]

Reguli de coerciție:

InputOptStr →OptFloat →
NoneNoneNone
[] / {}NoneNone
"RO123""RO123"None
123"123"123.0
"1.234,56""1.234,56"1234.56
"2,938.25 RON""2,938.25 RON"2938.25
["a", "b"]"a | b"primul număr valid

Pentru OptFloat, parser-ul detectează inteligent formatul european vs american:

if "," in s and "." in s:
    if s.rfind(",") > s.rfind("."):
        # format european: "1.234,56" → "1234.56"
        s = s.replace(".", "").replace(",", ".")
    else:
        # format american: "1,234.56" → "1234.56"
        s = s.replace(",", "")
elif "," in s:
    parts = s.split(",")
    if len(parts) == 2 and 1 <= len(parts[1]) <= 2:
        s = s.replace(",", ".")     # virgula = zecimale
    else:
        s = s.replace(",", "")      # virgula = mii
Avantajul de design: aplicația devine robustă la output-uri „dezordonate" ale modelului local fără să compromită rigorile validării. Coercer-ul nu inventează date — convertește doar reprezentarea.
