§ 1

Introducere și Motivație

Invoice AI Extractor este un sistem software complet care automatizează extragerea structurată a datelor din facturi PDF — atât digitale, cât și scanate — folosind modele de limbaj mari (LLM) de la OpenAI. Rezultatul este serializat în format XML compatibil cu sistemul contabil SAGA, gata de import direct.

Problema adresată este una practică și cu impact economic direct: procesarea manuală a facturilor consumă zeci de ore pe lună în orice organizație, este predispusă la erori umane și nu scalează. Soluțiile OCR tradiționale (bazate pe expresii regulate sau template-uri fixe) eșuează în fața diversității de formate.

§ 2

Arhitectura Generală a Sistemului

Aplicația urmează o arhitectură pipeline stratificată, cu separare clară a responsabilităților între module. Fiecare strat poate fi înlocuit sau extins independent, fără a afecta celelalte. Principalele trei straturi sunt:

Stratul de Prezentare app.py Interfața Streamlit: upload fișier, declanșarea pipeline-ului, afișare rezultate structurate, descărcare XML/JSON. Streamlit
Stratul de Business Logic src/*.py Procesare PDF, extragere AI, validare Pydantic, verificare firme, analiză risc, generare XML. Python 3.12+
Stratul de Integrare OpenAI API / ANAF / etc. Apeluri externe: LLM pentru extragere și căutare, API-uri publice pentru verificarea identității firmelor. REST API

Principii de proiectare

  • Graceful degradation — orice eșec al unui apel extern este capturat și returnat ca status="error"; pipeline-ul continuă.
  • Schema-first — toate datele trec prin modele Pydantic înainte de a fi consumate de alte module.
  • Paralelism controlat — verificarea furnizorului și a clientului rulează simultan (ThreadPoolExecutor(max_workers=2)) pentru a minimiza latența.
  • Fără state global — fiecare rulare este independentă; fișierele sunt salvate cu timestamp unic.

§ 5

Integrarea cu OpenAI API

Sistemul folosește OpenAI API în trei contexte distincte, fiecare cu un prompt engineering specific:

5.1 — Extragere date factură (modul text)

Când PDF-ul conține text digital suficient, acesta este trimis direct modelului GPT-4o prin chat.completions.create() cu response_format={"type": "json_object"}. Aceasta forțează modelul să returneze exclusiv JSON valid, eliminând riscul de răspunsuri cu text liber sau Markdown.

src/openai_extractor.py — extract_invoice_from_text() 
# temperature=0 → răspuns determinist, reproductibil
response = client.chat.completions.create(
    model=OPENAI_MODEL,
    temperature=0,
    response_format={"type": "json_object"},
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user",   "content": user_prompt},
    ],
)

5.2 — Extragere date factură (modul Vision)

Facturile scanate sunt convertite la imagini PNG (DPI 200, zoom ≈ 2.78×) și trimise ca payload multimodal cu image_url + detail: "high". Modelul vizualizează fiecare pagină și extrage datele cu aceeași schemă JSON.

src/openai_extractor.py — extract_invoice_from_images() 
# Payload multimodal: text instrucțiuni + N imagini base64
user_content = [
    {"type": "text", "text": instrucțiuni + JSON_SCHEMA},
    # pentru fiecare pagină PNG:
    {
        "type": "image_url",
        "image_url": {
            "url": f"data:image/png;base64,{b64}",
            "detail": "high",  # rezoluție maximă pt. text mic
        }
    }
]

5.3 — Verificare firmă prin web search

Pentru verificarea furnizorilor/clienților, sistemul folosește Responses API cu toolul web_search_preview. Modelul execută căutarea web intern și returnează un JSON structurat cu datele firmei din registrele naționale (ANAF, ONRC, etc.).

src/company_verifier.py — verify_company_openai() 
response = client.responses.create(
    model=OPENAI_MODEL,
    tools=[{"type": "web_search_preview"}],
    input=instruction,   # include country_hint + registries
)
⚠️ Fallback automat: dacă providerul primar (ANAF, OpenAPI.ro) eșuează din cauza WAF-ului sau a lipsei cheii API, sistemul comută automat pe backend-ul OpenAI web search, fără intervenția utilizatorului.

5.4 — Prompt engineering — principii aplicate

  • Schema explicită în prompt — structura JSON dorită este inclusă integral în fiecare mesaj, nu doar descrisă verbal.
  • Instrucțiune negativă — modelul este explicit instruit să nu inventeze date: "nu inventezi și nu deduci date care nu sunt scrise".
  • temperature=0 — elimină stochasticitatea răspunsului; același PDF → același JSON.
  • Separare system/user — rolul system stabilește identitatea asistentului; rolul user furnizează datele de intrare.
  • Null over hallucination — câmpurile lipsă sunt explicit null, nu estimate.
§ 6

Validarea Datelor cu Pydantic

Pydantic v2 joacă rolul unui contract de date între modulul de extragere AI și restul sistemului. Toate câmpurile sunt Optional — extragerea parțială este un comportament normal, nu o eroare.

src/schema.py — structura InvoiceData (simplificat) 
class InvoiceData(BaseModel):
    model_config = ConfigDict(extra="ignore")  # câmpuri extra ignorate

    invoice_number: Optional[str] = None
    invoice_date:   Optional[str] = None
    currency:       Optional[str] = None

    supplier: Supplier      = Field(default_factory=Supplier)
    customer: Customer      = Field(default_factory=Customer)
    items:    List[InvoiceItem] = Field(default_factory=list)
    totals:   Totals         = Field(default_factory=Totals)

    # Extensia de verificare (adăugată post-extragere)
    supplier_verification: Optional[CompanyVerification] = None
    supplier_risk_analysis: Optional[CompanyRiskAnalysis] = None

Ierarhia modelelor de date

ModelCâmpuri cheieUtilizare
Suppliername, tax_id, registration_number, address, iban, bankDate furnizor din factură
Customername, tax_id, registration_number, addressDate client din factură
InvoiceItemdescription, quantity, unit_price, vat_rate, net_amount, vat_amount, gross_amountLinie din factură
Totalssubtotal, vat_total, grand_totalTotaluri factură
CompanyVerificationverified, status, official_name, tax_id, vat_status, company_status, caen_codeRezultat verificare registru

✅ ConfigDict(extra="ignore") pe fiecare model asigură că un răspuns OpenAI cu câmpuri suplimentare neașteptate nu aruncă excepție — sistemul ignoră silențios câmpurile necunoscute.
§ 7

Generarea XML compatibil SAGA

Generarea XML este deterministă și locală — nu implică niciun apel AI. Folosește xml.etree.ElementTree pentru construcția arborelui și xml.dom.minidom pentru formatarea indentată a output-ului final.

Structura documentului XML generat

Exemplu output XML (structură) 
<?xml version="1.0" encoding="utf-8"?>
<Invoice>
  <InvoiceNumber>DVN-23</InvoiceNumber>
  <InvoiceDate>2026-05-04</InvoiceDate>
  <Currency>RON</Currency>
  <Supplier>
    <Name>...</Name>
    <TaxID>RO12345678</TaxID>
    <IBAN>RO49 AAAA ...</IBAN>
  </Supplier>
  <Items>
    <Item>
      <Description>Servicii consultanță</Description>
      <Quantity>1</Quantity>
      <VATRate>19</VATRate>
      <GrossAmount>1190</GrossAmount>
    </Item>
  </Items>
  <SupplierVerification>
    <Verified>true</Verified>
    <RiskScore>10</RiskScore>
    <RiskLevel>low</RiskLevel>

Funcția safe_text() normalizează toate valorile înainte de inserție: None → "", float → 4 zecimale fără zerouri finale, bool → "true"/"false". Caracterele speciale XML (< > &) sunt escapate automat de ElementTree.

§ 8

Verificarea Firmelor prin API-uri Externe

Modulul company_verifier.py implementează un dispatcher multi-provider cu fallback automat. Providerul se configurează prin variabila de mediu COMPANY_API_PROVIDER.

ProviderEndpointAutentificareObservații
openai Responses API + web_search_preview OPENAI_API_KEY Funcționează de pe orice rețea; caută ANAF, ONRC, listafirme.ro, termene.ro; suport multi-țară
anaf webservicesp.anaf.ro/api/.../v9/ws/tva public (opțional token) API public ANAF; detectare WAF block; payload POST cu CUI + dată
openapi api.openapi.ro/v1/companies/{cif} x-api-key header Serviciu comercial; răspuns bogat cu CAEN, status, adresă
listafirme listafirme.ro/api/info-firma.asp COMPANY_API_KEY param Serviciu comercial românesc; fallback pentru OpenAPI.ro

Logica de fallback

src/company_verifier.py — verify_company() — fallback logic 
# Dacă providerul primar eșuează AND avem OPENAI_API_KEY:
if (
    provider != "openai"
    and primary.status in ("error", "not_configured")
    and OPENAI_API_KEY
):
    fallback = verify_company_openai(company_name, tax_id)
    if fallback.verified or fallback.status == "not_found":
        return fallback  # înlocuiește rezultatul primar

Normalizarea CUI-ului românesc

Funcția normalize_tax_id() elimină spațiile și prefixul opțional RO (folosit pentru firmele plătitoare de TVA), returnând codul numeric pur necesar API-urilor.

Compararea câmpurilor factură ↔ registru

Funcția compare_invoice_supplier_with_verified_data() compară:

  • Nume — comparație fuzzy la nivel de slug (doar alfanumerice, lowercase), cu relație de subșir
  • Tax ID — comparație exactă după normalizare
  • Adresă — overlap de tokeni ≥ 1/3 din minimum (rezistentă la diferențe de format)
§ 9

Analiza Euristică de Risc

Modulul company_risk_analyzer.py calculează un scor de risc 0–100 din patru semnale independente, agregate aditiv. Scorul final este binarizat în trei niveluri:

SemnalPenalizareCondiție
Firma neverificată +30 pct. status ∉ {verified, not_configured, disabled}
Status juridic inactiv +50 pct. company_status conține: radiat, suspendat, lichidare, faliment…
Tax ID nepotrivit +40 pct. tax_id din factură ≠ tax_id din registru (după normalizare)
Nume nepotrivit +20 pct. Slug-ul numelui din factură nu se potrivește cu registrul
Mențiune negativă online +10 pct. / articol titlu/snippet conține termeni din vocabularul de risc

§ 10

Interfața Web — Streamlit

Interfața utilizator este construită cu Streamlit 1.30+ și extinsă cu CSS custom injectat prin st.markdown(..., unsafe_allow_html=True). Design-ul urmărește un registru editorial tipografic (tipăritură academică digitală).

Componente de UI

  • Title page — antet universitar cu logo-uri UPB și FAIMA (base64 embeds)
  • Bibliographic strip — autor, coordonator, locație/dată apărare
  • File uploader — st.file_uploader(type=["pdf"])
  • Progress bar — 8 etape cu mesaje descriptive (0% → 100%)
  • Run metadata strip — metodă extragere, câmpuri detectate, elemente, timp total
  • Tabs — XML output · Structured data (JSON) · Source text · Company verification
  • Download buttons — XML și JSON descărcabile direct
  • Risk bands — CSS cards cu culori semantice (green/yellow/red)
  • Online mentions — carduri cu tip (website/registry/social/news), domeniu, dată, snippet

§ 12

Concluzii Tehnice

Sistemul demonstrează că un pipeline hibrid PDF → LLM → XML poate înlocui cu succes procesarea manuală a facturilor, oferind simultan:

  • Robustețe față de diversitatea formatelor — același sistem procesează atât facturi digitale (text layer) cât și scanate (Vision multimodal), fără configurare per-template.
  • Transparență — fiecare decizie de clasificare a riscului este explicată prin mesaje de warning specifice, nu doar un scor opac.
  • Extensibilitate — adăugarea unui nou provider de verificare firmă necesită o singură funcție nouă și un entry în dispatcherul verify_company().
  • Reziliență — nicio eroare externă nu blochează pipeline-ul; toate eșecurile sunt gestionate și raportate în UI fără a pierde rezultatele deja computate.
  • Reproductibilitate — temperature=0 și schema strictă JSON asigură că același PDF produce același output la rulări repetate.

Limitări cunoscute și direcții de dezvoltare

  • Scorul de risc este euristic — nu are validare statistică pe un dataset real de facturi frauduloase.
  • Costul per factură depinde de lungimea textului și de numărul de apeluri API (2–4 apeluri OpenAI/factură).
  • Suportul pentru facturi în limbi non-europene poate necesita ajustări de prompt.
  • Integrarea directă cu SAGA (import automat XML) ar elimina pasul manual de import.
🎓 Proiectul contribuie cu o abordare aplicată a LLM-urilor în automatizarea proceselor de business, combinând ingineria prompt-urilor, validarea formală a datelor și integrarea cu registre publice românești — o intersecție mai puțin explorată în literatura de specialitate curentă.
